## TL;DR

The original plan was to **scrape `claude.ai/settings/usage` inside a hidden
`WKWebView`**. That approach crashed hard and repeatedly on this machine, and
no amount of window/threading trickery fixed it. We pivoted to the design the
spec actually called for: **read usage from `api.anthropic.com/api/oauth/usage`
over plain `URLSession`, authenticated with our own OAuth token**. That works,
is stable, and has no WebKit dependency at all.

## Phase A — The WKWebView dead end

### What we tried
A background `WKWebView` would load the usage page, we'd inject JavaScript to
read the "Current session" / "All models" DOM nodes, and post the numbers back
via a `WKScriptMessageHandler`.

### What went wrong
Persistent `EXC_BAD_ACCESS` crashes in `objc_release` on the main thread. The
Console logs were the smoking gun:

```
Error acquiring assertion: ... doesn't have entitlement
    com.apple.runningboard.assertions.webkit
Ignoring request to entangle context after pre-commit
Entangling fence requested after pre-commit
```

The WebKit **GPU/compositing process** could not acquire its RunningBoard
assertions, and the render pipeline crashed. `com.apple.runningboard.assertions.webkit`
is a **private Apple entitlement** — not something a self-signed dev app can grant.

### Things that did *not* fix it
- Moving the web view off-screen (`-10000, -10000`) — still crashed.
- A separate `WKWebView` for the login window — still crashed.
- A 1×1 transparent on-screen window — survived **launch** (nothing composited),
  but any *visible* web view (the login window) crashed the moment it rendered.

### Why URLSession scraping didn't work either
Before giving up on the web entirely, we tried `URLSession` against
`claude.ai/settings/usage`:
- The page is a **pure SPA** — a ~5.5 KB HTML shell with `<div id="root">`.
  All usage data is rendered client-side after hydration, so there's nothing to
  parse in the raw HTML.
- The claude.ai cookie-based API endpoints returned **403
  `account_session_invalid`** — they need in-memory/IndexedDB auth state that
  only exists inside the live web app, not something cookies alone provide.

**Conclusion:** anything WebKit-based is a non-starter on this setup, and
cookie-scraping the SPA is a dead end.

## Phase B — The pivot to OAuth + the real API

The project spec described the *right* design all along: read the usage API
directly. The question was just **how to authenticate**.

### Attempt 1: read Claude Code's Keychain token
The spec said Claude Code stores `{ "claudeAiOauth": { "accessToken": ... } }`
in the Keychain under service `Claude Code-credentials`. On this machine that
item existed but contained **only MCP plugin OAuth state** — no user token.
Dead end for *this* environment, and fragile across Claude Code versions anyway.

### Attempt 2 (the winner): our own OAuth PKCE flow
Instead of borrowing Claude Code's token, Claudicator now runs the **same
sign-in flow `claude setup-token` uses** and gets its *own* token.

We pulled the exact constants straight out of the installed Claude Code bundle
(`cli.js`) rather than guessing:

```
CLIENT_ID:  9d1c250a-e61b-44d9-88ed-5944d1962f5e
AUTHORIZE:  https://claude.ai/oauth/authorize   (?code=true → shows a paste code)
TOKEN:      https://console.anthropic.com/v1/oauth/token
REDIRECT:   https://console.anthropic.com/oauth/code/callback
SCOPES:     org:create_api_key user:profile user:inference
```

The flow (manual "paste-code" PKCE):
1. Generate a PKCE verifier/challenge, open the browser to the authorize URL.
2. User logs in and approves **in their own browser** — the app never sees a
   password.
3. The callback page shows a `code#state` string; the user pastes it back.
4. The app exchanges it at the token endpoint for an access + refresh token,
   stored in the Keychain under Claudicator's own service name.
5. Usage requests use that token with header `anthropic-beta: oauth-2025-04-20`.

This is self-contained, doesn't depend on Claude Code's storage, and keeps the
password boundary clean.

## Phase C — Two stability bugs after it worked

Usage data showed up correctly, but two issues remained.

### 1. The MenuBarExtra freeze / stack overflow
Opening the popover froze the app with `EXC_BAD_ACCESS (code=2)` at a **stack
address** — classic runaway recursion. The cause: a **1-second countdown timer**
mutated `@Published` state every second, and while the `MenuBarExtra` popover
was open this drove SwiftUI into recursive menu reconciliation.

**Fix:** stop publishing every second. The store now keeps the quota **reset
dates**; the live "Resets in …" string is rendered locally by a
`TimelineView(.periodic(by: 60))` in `UsageRowView`. `TimelineView` only
re-renders its own subtree — it never republishes the store — so the menu stays
stable. (We also dropped the progress-bar `.animation`, another in-popover
reconciliation risk.)

### 2. Endless Keychain password prompts
The app hit the Keychain on **every** 90-second poll (twice, even). Each access
can trigger an OS prompt.

**Fix:** `OAuthService` now caches the tokens in memory after the first read, so
the Keychain is touched **once per launch** (plus the rare token-refresh write).

> Note: across *rebuilds*, macOS may still re-prompt, because the "Always Allow"
> decision is bound to the app's code signature and Xcode re-signs every build.
> Setting a stable signing Team makes the permission persist.

## Where it landed

A stable Phase 1:

- **No WebKit anywhere.** Pure `URLSession` + `Security` + `CryptoKit`.
- **Self-contained OAuth** sign-in (browser authorize → paste code).
- **In-memory token cache**, Keychain-backed.
- **`TimelineView` countdown** instead of a per-second republish.
- **`os.Logger` observability** (subsystem `com.ariross.claudicator`).

### File map
| File | Role |
|---|---|
| `ClaudicatorApp.swift` | `MenuBarExtra` entry point |
| `Models/UsageResponse.swift` | API response model (`five_hour`, `seven_day`, …) |
| `Services/OAuthService.swift` | PKCE flow, token exchange/refresh, in-memory cache |
| `Services/KeychainService.swift` | `OAuthTokens` model + Keychain read/write |
| `Services/UsageStore.swift` | Polling, fetch, published state |
| `Services/Log.swift` | Unified logging |
| `Views/ContentView.swift` | Popover layout |
| `Views/UsageRowView.swift` | Progress bar + `TimelineView` countdown |
| `Views/ConnectView.swift` | Connect window + flow (no WebKit) |

## Lessons for the next maintainer

1. **Don't reach for `WKWebView`** in this app. The GPU-process entitlement
   issue is environmental and not fixable from app code.
2. **Authenticate with the OAuth PKCE flow**, not by borrowing Claude Code's
   Keychain item — the latter's format and contents vary.
3. **Never mutate `@Published` state on a fast timer while a `MenuBarExtra`
   popover can be open.** Use `TimelineView` for anything that ticks.
4. **Cache secrets in memory.** Touching the Keychain in a hot path means
   password prompts.
5. The OAuth constants live in the Claude Code `cli.js` bundle — grep there if
   they ever change.
